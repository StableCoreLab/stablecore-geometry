# Box2 开发设计文档

## 1. 文档目的

本文档用于定义几何库中二维包围盒 `Box2<T>` 的开发方案。后续开发人员必须按本文档实现，不再按临时理解扩展。

本文档覆盖以下内容：

- `Box2<T>` 的类型定位
- 数据表达与不变量
- 有效盒、退化盒、空盒规则
- 公共接口边界
- 与线段、轮廓、算量场景的关系
- 第一版实现范围

本文档是 `library-design.md`、`point-vector-design.md`、`segment-design.md` 在包围盒层的实现依据。

## 1.1 文档关系

本文档与其它设计文档的关系如下：

- `library-design.md` 负责定义 `Box2<T>` 必须尽早进入核心类型，以及它在 `types` 层中的职责边界
- `segment-design.md` 负责定义所有边段都必须提供 `GetBoundingBox()`，并要求其返回语义与 `Box2<T>` 保持一致
- `point-vector-design.md` 负责定义 `Point2<T>` 的语义边界，而 `Box2<T>` 的 `min/max` 正建立在该点类型之上

执行要求：

- `Box2<T>` 的语义不得脱离总设计单独漂移
- `Segment2<T>` 的包围盒接口不得另起一套返回语义
- 若未来修改 `Box2<T>` 的有效性或边界规则，必须同步检查 `segment-design.md` 与总设计文档

## 2. 设计定位

第一版 `Box2<T>` 只表示二维轴对齐包围盒。

也就是说：

- 它是 AABB
- 它不是任意旋转矩形
- 它不是业务语义上的“房间矩形”或“构件矩形”
- 它首先服务粗判、过滤、索引和范围表达

对未来算量产品来说，`Box2<T>` 主要承担以下职责：

- 图元粗判
- 候选对象快速过滤
- 空间索引基础单元
- 边段、轮廓、区域的包围范围表达
- 算量预筛时的快速排除

因此它的设计目标必须是：

- 结构简单
- 语义稳定
- 易于组合
- 对退化情况可控

## 3. 总体设计结论

第一版统一按以下命名实现：

```cpp
template <typename T>
struct Box2;

using Box2d = Box2<double>;
using Box2i = Box2<int>;
```

`Box2<T>` 表示二维轴对齐闭包围盒。

执行结论：

- `Box2<T>` 是核心基础类型，放在 `types` 层
- 它表达数据和基础不变量
- 复杂几何关系判断仍放在 `algorithm` 层

## 4. 为什么 Box2 必须尽早进入核心类型

`Box2<T>` 不是附属工具，而是后续工程几何流程中的基础设施。

原因：

- 它是大多数粗判逻辑的第一道过滤
- 线段求交、区域求交、裁剪都会依赖它做候选收缩
- 轮廓和区域一旦做大，没有包围盒就无法高效筛选
- 算量里经常需要先做快速范围过滤，再进入精算

如果 `Box2<T>` 不尽早统一，后续常见问题会很快出现：

- 每个类型自己写一套包围范围表达
- 粗判逻辑散落在各模块
- 相交与过滤接口难以统一
- 退化规则和边界语义混乱

因此 `Box2<T>` 必须视为核心类型，不应等到后面再补。

## 5. 类型语义

### 5.1 `Box2<T>` 只表示轴对齐包围范围

`Box2<T>` 表示：

- x 方向区间
- y 方向区间

它的几何意义是二维轴对齐闭区域：

- `min.x <= x <= max.x`
- `min.y <= y <= max.y`

### 5.2 `Box2<T>` 不是一般矩形对象

`Box2<T>` 不承担以下语义：

- 任意角度旋转矩形
- 面向绘图的矩形图元
- 有方向的矩形边界
- 业务语义对象

如果未来需要这些对象，应另建类型，不允许把 `Box2<T>` 泛化成“万能矩形”。

### 5.3 `Box2<T>` 是闭区间语义

第一版统一采用闭区间规则。

也就是说：

- 边界点属于盒内
- 顶点属于盒内
- 退化到线段或点时，该对象仍可被盒表达

这条规则对工程过滤更稳定，也更适合和边段端点、交点等结果配合。

## 6. 数据表达与不变量

### 6.1 第一版核心数据

`Box2<T>` 第一版必须至少保存：

- `min` 点
- `max` 点

推荐形式：

```cpp
template <typename T>
struct Box2
{
    Point2<T> min{};
    Point2<T> max{};
};
```

### 6.2 不变量

对有效盒，必须满足：

- `min.x <= max.x`
- `min.y <= max.y`

### 6.3 构造规则

第一版统一允许以下构造方式：

- 默认构造
- `Box2(minPoint, maxPoint)`

但必须明确：

- “能构造”不代表“有效”
- 默认构造必须产生无效盒状态，而不是隐式变成有效点盒

执行要求：

- 默认构造后的 `IsValid()` 必须返回 `false`
- 当前版本不允许把默认构造语义实现成 `min == max == Point2<T>{}`
- 默认构造用于表达“尚未建立范围”的初始状态

### 6.4 不允许自动排序

第一版不允许在 `(minPoint, maxPoint)` 构造中自动偷偷交换分量。

原因：

- 会掩盖上层错误输入
- 让调用方误以为自己传入的是已规范化数据
- 对调试不利

第一版统一做法是：

- 原样存储
- 通过 `IsValid()` 暴露问题
- 若确实需要规范化盒，提供显式工厂或显式规范化接口

执行结论：

- 构造函数不负责静默修正错误顺序
- 有效性检查必须显式存在

## 7. 有效盒、退化盒、空盒规则

### 7.1 有效盒

满足以下条件的盒为有效盒：

- `min.x <= max.x`
- `min.y <= max.y`

### 7.2 退化盒

以下情况仍属于有效盒：

- `min.x == max.x` 且 `min.y < max.y`，退化为竖线段范围
- `min.x < max.x` 且 `min.y == max.y`，退化为横线段范围
- `min.x == max.x` 且 `min.y == max.y`，退化为点范围

执行结论：

- 退化盒是有效盒
- 第一版不把退化盒当异常

原因：

- 点、线段、弧段的包围盒天然可能退化
- 算量和几何粗判都必须接受这种情况

### 7.3 空盒

第一版把空盒定义为“无效盒”，即：

- 只要存在 `min.x > max.x` 或 `min.y > max.y`
- 就视为无效盒

也就是说，第一版不额外引入与“无效盒”分离的“有效但空”状态。

这样做的好处是：

- 状态更少
- 基础类型更简单
- 和包围盒作为几何范围的直觉更一致

### 7.4 为什么不把零面积盒视为空盒

零面积不代表空。

例如：

- 一个点的包围盒
- 一条竖线段的包围盒
- 一条横线段的包围盒

都可能面积为零，但它们仍有明确几何意义。

所以：

- `Area() == 0` 不等于空盒
- 是否为空，应由有效性规则决定，而不是由面积决定

## 8. 基础公共接口

`Box2<T>` 第一版必须至少提供以下接口。

### 8.1 有效性接口

- `IsValid()`

语义：

- 判断 `min/max` 顺序是否满足盒语义

调用要求：

- 所有尺寸、中心点、包含、相交类逻辑都应以有效盒为前置条件

### 8.2 尺寸接口

- `Width()`
- `Height()`

执行要求：

- 对有效盒，返回非负值
- 对无效盒，不允许静默返回伪结果

执行结论：

- 将有效盒作为调用前置条件
- 调试期可以断言，但不得把无效输入悄悄修正成有效结果

### 8.3 面积接口

- `Area()`

统一语义：

- 有效盒面积为 `Width() * Height()`
- 退化盒面积允许为零

调用要求：

- 仅对有效盒调用

### 8.4 顶点接口

必须提供：

- `GetMinPoint()`
- `GetMaxPoint()`

实现方式可以直接公开 `min/max` 字段，也可以只通过访问接口暴露，但当前版本对外语义必须清楚：

- `min` 是左下意义上的最小坐标点
- `max` 是右上意义上的最大坐标点
- 对外必须稳定提供 `GetMinPoint()` 与 `GetMaxPoint()`

### 8.5 中心点接口

必须提供：

- `Center()`

该接口属于包围盒自身的自然属性，可放在类型层。

调用要求：

- 仅对有效盒调用

### 8.6 扩张接口

必须提供：

- `ExpandToInclude(point)`
- `ExpandToInclude(box)`

原因：

- 包围盒在工程中非常常见的用法是累积构造
- 线段、轮廓、点集的整体范围都需要这类能力

扩张语义必须统一如下：

- 若当前盒无效，`ExpandToInclude(point)` 后必须变成以该点为 `min/max` 的有效点盒
- 若当前盒无效，`ExpandToInclude(validBox)` 后必须变成该有效盒
- 若传入盒无效，`ExpandToInclude(invalidBox)` 必须不改变当前对象
- 若当前盒与传入对象都有效，则结果必须为同时包含两者的最小轴对齐盒
- `ExpandToInclude` 只允许扩张，不允许缩小，也不允许把有效盒变成无效盒

### 8.7 当前版本不提供规范化接口

当前版本不提供自动排序后的规范化接口，例如：

- `MakeNormalized()`

如果后续确实需要，应单独补设计文档后再引入。

执行要求：

- 不允许把修正行为偷偷塞进普通构造函数
- 不允许在当前版本中引入未定规则的规范化接口

## 9. 禁止放入类型层的内容

`Box2<T>` 不应直接承载以下复杂逻辑：

- 带容差的包含判断
- 带容差的盒相交判断
- 距离算法
- 裁剪算法
- 空间索引结构
- 业务层过滤规则

原因：

- 这些都属于规则性算法，不是纯数据语义
- 很多逻辑会依赖容差，必须统一放在算法层

## 10. 比较规则

### 10.1 严格比较

`operator==` 与 `operator!=` 若实现，只表示严格分量比较。

也就是说：

- `min` 严格相等
- `max` 严格相等

### 10.2 容差比较

不允许把容差比较写进 `operator==`。

如需近似判断，应走独立接口，例如未来可定义：

- `IsEqual(lhs, rhs, eps)`

这与 `Point2<T>`、`Vector2<T>`、`Segment2<T>` 的规则保持一致。

## 11. Box2 与点、线段的关系

### 11.1 点到盒的关系

点本身不是盒，但：

- 点可以退化成一个点盒

也就是说：

- `min == max == point`

是合法且有效的 `Box2<T>`。

### 11.2 线段到盒的关系

所有有限边段都必须能稳定生成一个 `Box2<T>`。

对 `LineSegment2<T>`：

- 包围盒由起终点坐标直接确定

对 `ArcSegment2<T>`：

- 包围盒不能只看起终点
- 必须考虑是否跨越圆弧坐标极值角

这一条与 `segment-design.md` 中“圆弧包围盒必须是几何正确包围盒”的约束是同一条规则，不允许两份文档分别演化出不同解释。

### 11.3 轮廓和区域对盒的依赖

未来 `Polyline2`、`Polygon2`、区域边界层都会依赖 `Box2<T>` 做：

- 快速粗判
- 局部候选收缩
- 预过滤

所以 `Box2<T>` 的退化规则和边界语义必须从现在开始固定。

## 12. 面向算量场景的设计要求

算量产品里，`Box2<T>` 的核心价值通常不是“表达一个矩形”，而是：

- 快速判定是否值得继续精算
- 先过滤远离目标的候选构件
- 为局部求交、局部测距缩小问题规模

因此 `Box2<T>` 的设计必须优先满足：

- 可快速构造
- 可稳定合并
- 可接受退化图元
- 语义简单，不隐藏修正逻辑

### 12.1 为什么要允许退化盒

因为很多工程对象天然会产生退化包围盒：

- 单个点
- 共线点集
- 极短边段
- 退化轮廓片段

如果把这些都视为异常，后续算量预处理会很难做。

### 12.2 为什么不把 Box2 做成复杂算法中心

因为算量真正复杂的地方不在盒本身，而在：

- 盒与图元的关系判断
- 盒参与的裁剪
- 盒参与的空间组织

这些都应留在算法和上层结构中，而不是塞进 `Box2<T>` 类型本身。

## 13. 后续算法接口方向

本文件不直接定义算法实现，但明确约束后续算法命名方向。

例如未来可有：

- `Contains(box, point, eps)`
- `Contains(boxA, boxB, eps)`
- `Intersects(boxA, boxB, eps)`
- `Distance(point, box)`
- `Distance(segment, box)`

执行原则：

- 复杂判断与容差相关逻辑放在 `algorithm` 层
- `Box2<T>` 本身保持轻量
- 算法层默认只接受有效盒；若要处理无效盒，必须显式定义返回策略

## 14. 第一版不做的内容

第一版 `Box2<T>` 不做以下内容：

- 旋转包围盒
- 三维盒
- 盒裁剪算法全集
- 空间树结构
- 面向业务的区域框选语义
- 与 GUI 相关的显示矩形能力

第一版严格限制为：

- 二维轴对齐包围盒
- 数据表达
- 基础尺寸与扩张能力
- 为后续算法预留统一语义

## 16. 开发验收标准

后续实现完成后，必须满足以下标准：

- `Box2<T>` 明确表示二维轴对齐包围盒
- 有效盒与无效盒规则清楚
- 退化盒被视为有效盒
- 默认构造不会偷偷假装成规范有效盒
- 构造函数不静默修正错误顺序
- `Segment2<T>` 可以稳定返回 `Box2<T>` 风格的包围盒
- 类型层保持轻量，不混入复杂算法

## 17. 最终执行结论

后续 `Box2<T>` 开发必须按以下原则执行：

- `Box2<T>` 只表示二维轴对齐闭包围盒
- 核心数据采用 `min/max` 两点表达
- 有效盒满足 `min <= max`
- 退化盒是有效盒，不是异常
- 无效盒用于表达错误顺序或空状态
- 不在构造阶段偷偷修正输入
- 复杂关系判断留在 `algorithm` 层
- 设计优先服务粗判、过滤、索引和算量预筛场景

文档闭环要求：

- 总体目标与模块边界以 `library-design.md` 为总入口
- `Segment2<T>` 的包围盒接口约束以 `segment-design.md` 与本文档共同收敛
- 任何涉及 `Box2<T>` 语义的后续实现或文档修改，都必须同步检查上述文档是否需要联动调整

本文档即为 `Box2<T>` 的实现依据。
