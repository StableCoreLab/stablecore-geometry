# StableCore Geometry 设计方案

## 1. 目标定位

这个仓库不是做通用数学库，也不是做偏学术的计算几何实验集，而是要成为后续算量软件的几何算法基础库。

因此它的目标应当是：

- 面向工程场景，而不是面向竞赛题
- 先解决稳定、可测、可维护，再追求算法花哨
- 先把 2D 基础打牢，再考虑 3D
- 让几何类型、几何谓词、几何运算之间职责清楚

对未来算量软件来说，这个库最终要支撑的不是单一“点线面计算”，而是一整套工程几何处理流程，例如：

- 构件轮廓表达
- 线段、折线、区域关系判断
- 投影、裁剪、求交、测距
- 面积、长度、周长、中心点等基础量计算
- 为后续图形清洗、轮廓拼接、算量规则计算提供几何底座

## 2. 设计原则

### 2.1 先做工程几何内核，不急着做“大而全”

第一阶段不要把目标放在“支持所有几何对象”。应先确保以下能力可靠：

- 点和向量
- 线段
- 包围盒
- 容差比较
- 投影
- 距离
- 相交

这些是几乎所有后续算法的基础。

### 2.2 类型层和算法层分离

类型负责表达数据，算法负责表达行为。

不建议一开始就把大量算法塞进类型成员函数里。原因很直接：

- 类型会变重
- 接口边界会混乱
- 测试粒度不清晰
- 以后扩到 `Point2<T>`、`Point3<T>` 会更难演进

建议：

- `types` 只放几何对象和少量显然成立的小函数
- `algorithm` 放距离、投影、相交、面积、包含等算法
- `common` 放容差和基础数值工具

### 2.3 容差是全库基础设施

工程几何和理想数学不一样。以后只要涉及：

- 共线
- 平行
- 点在线段上
- 相交
- 闭合

就一定会遇到浮点误差问题。

所以容差不能是“零散的 if 判断”，而必须是全库统一规则。

结论：

- 不允许在算法里散落 `1e-6`、`1e-9`
- 所有几何判断都必须经过统一容差接口
- 文档里要区分“拓扑相等”和“数值近似相等”

### 2.4 复杂算法返回结果结构体

例如线段求交，调用方通常不只关心“是否相交”，还关心：

- 是否平行
- 是否共线
- 是否是唯一交点
- 交点位置
- 参数位置是否在端点上

因此复杂算法不能只返回 `bool`，而应该返回结果结构体。

这条原则后续对算量很重要，因为算量规则通常需要过程信息，而不是只有最终真假。

### 2.5 API 命名要表达工程语义

命名不要图省事。

例如：

- `ProjectPointToSegment` 优于 `Project`
- `IsPointOnSegment` 优于 `Contains`
- `SegmentIntersectionResult` 优于 `Result`

后面库规模一大，命名是否自解释，会直接决定维护成本。

## 3. 库的职责边界

这个几何库建议只负责“几何层”，不要一开始混入业务层。

应该负责：

- 基础几何对象
- 基础几何关系判断
- 基础几何运算
- 稳定、可复用的结果表达

不应该负责：

- 工程构件业务语义
- 楼层、墙、梁、板等业务对象
- 算量规则解释
- 文件解析
- 图形界面

这样库才能长期稳定，不会被上层业务拖乱。

## 4. 推荐分层

建议沿用当前仓库里已经有的目录思路：

```text
include/
  common/
  types/
  algorithm/

src/
tests/
docs/
```

各层职责如下。

### 4.0 当前专项设计文档

总设计文档负责定义全局目标、分层和演进边界。

当前已经拆分出的专项设计文档如下：

- `point-vector-design.md`：负责 `Point2<T>` 与 `Vector2<T>` 的类型语义、运算符边界和容差接口边界
- `segment-design.md`：负责 `Segment2<T>`、`LineSegment2<T>`、`ArcSegment2<T>` 的统一边段抽象与参数化规则
- `box-design.md`：负责 `Box2<T>` 的包围盒语义、有效性规则和基础接口
- `polyline-design.md`：负责 `Polyline2<T>` 的有序连续路径语义、闭合语义、整体参数化和顶点访问语义
- `polygon-design.md`：负责 `Polygon2<T>` 的区域语义、外环/洞环结构、面积语义和方向规则
- `sdk-design.md`：负责未来 `DLL + lib + headers` 发布形态下的模板边界、SDK 导出边界和 ABI 收敛策略
- `export-design.md`：负责未来 DLL 导出宏、显式实例化、SDK 头文件组织和复杂对象 ABI 暴露边界

执行要求：

- 总设计文档给出全局边界
- 专项设计文档给出可直接落地实现的细化规则
- 若总设计与专项文档冲突，应先回到总设计统一修订，再同步修改专项文档

### 4.1 `common`

作用：

- 容差配置
- 基础比较函数
- 标量数学辅助

这个层要尽量纯，不依赖高层几何对象。

建议未来包含：

- `Epsilon`
- `NumericCompare`
- `MathUtils`

### 4.2 `types`

作用：

- 定义点、向量、线段、包围盒等基础对象

建议未来包含：

- `Point2<T>`
- `Vector2<T>`
- `Segment2<T>`
- `Box2<T>`
- `Polyline2<T>`
- `Polygon2<T>`

这个层重点是“表达数据”和“保存不变量”。

当前执行落点：

- `Polyline2<T>` 的具体实现边界以 `polyline-design.md` 为准
- `Polygon2<T>` 的具体实现边界以 `polygon-design.md` 为准

### 4.3 `algorithm`

作用：

- 对基础几何对象做运算与判断

建议分模块，而不是所有算法都堆在一个头文件里。

初期建议：

- `Predicate2d`
- `Distance2d`
- `Project2d`
- `Intersect2d`
- `Measure2d`

后续可扩展：

- `Polyline2d`
- `Polygon2d`
- `Clip2d`
- `Topology2d`

当前执行落点：

- `Polyline2<T>` 属于阶段二“轮廓和区域基础”的路径层能力
- `Polygon2<T>` 属于阶段二“轮廓和区域基础”的区域层能力

## 5. 类型系统设计

### 5.1 `Point2<T>` 和 `Vector2<T>`

你已经明确希望用模板类统一 `Point2d`、`Vector2d`、`Point2i`、`Vector2i`，这个方向是合理的。

建议基础形式为：

- `Point2<T>`
- `Vector2<T>`

类型别名为：

- `Point2d`
- `Point2i`
- `Vector2d`
- `Vector2i`

这样做的好处：

- 避免重复定义结构
- 便于未来扩展到 `Point3<T>`、`Vector3<T>`
- `double` 和 `int` 场景可以共用统一语义

但要注意一个边界：

- 算法层大多数核心计算仍建议以 `double` 为主

原因是几何算法里，整数类型多适合作为输入表达，真正计算时通常需要浮点。

也就是说：

- 类型模板化是合理的
- 算法不必为了“泛型”而牺牲可控性

当前执行落点：

- `Point2<T>` 与 `Vector2<T>` 的具体实现边界以 `point-vector-design.md` 为准

### 5.2 `Segment2<T>`

建议不要只定义 `Segment2d`，而是从一开始就考虑：

- `Segment2<T>`

别名：

- `Segment2d`
- `Segment2i`

虽然工程算法主战场还是 `Segment2d`，但模板化后整个类型系统更统一。

当前执行落点：

- `Segment2<T>` 体系的具体实现边界以 `segment-design.md` 为准

### 5.3 `Box2<T>`

包围盒建议尽早进入核心类型。

原因：

- 它是后续加速和粗判的基础
- 线段求交、图元裁剪、区域求交都会用到
- 算量里经常需要快速过滤候选对象

当前执行落点：

- `Box2<T>` 的具体实现边界以 `box-design.md` 为准
- `Segment2<T>` 的 `GetBoundingBox()` 语义必须与 `box-design.md` 保持一致

### 5.4 折线和多边形暂缓进入第一阶段

`Polyline2<T>` 和 `Polygon2<T>` 很重要，但不建议一开始就做。

因为一旦引入，就会立刻牵出：

- 闭合规则
- 顶点重复清理
- 自交判断
- 方向约定
- 洞和外环

这些复杂度明显高于点线段阶段。

当前执行落点：

- `Polyline2<T>` 的语义细节以 `polyline-design.md` 为准
- `Polygon2<T>` 的语义细节以 `polygon-design.md` 为准

## 6. 算法模块设计

### 6.1 `Predicate2d`

这是全库真正的地基。

建议包含：

- `IsZero`
- `IsEqual`
- `IsLess`
- `IsGreater`
- `Dot`
- `Cross`
- `Orientation`
- `IsParallel`
- `IsCollinear`
- `IsPointOnSegment`

其中最关键的是：

- 方向判断
- 共线判断
- 点在线段上判断

因为它们会被几乎所有后续算法复用。

### 6.2 `Distance2d`

建议第一阶段至少支持：

- 点到点距离
- 点到线段距离
- 点到包围盒距离
- 线段到线段距离

后面很多算量规则，本质上都是从这些基础距离组合出来的。

### 6.3 `Project2d`

建议支持：

- 点投影到直线
- 点投影到线段

返回结果不应只有投影点，还应包含：

- 投影参数
- 距离平方
- 是否落在线段内部

因为这些信息在后续判断“最近点是否有效”时很常用。

### 6.4 `Intersect2d`

第一阶段先把线段求交做稳。

建议结果至少能区分：

- 不相交
- 唯一交点
- 平行不交
- 共线重叠

如果以后要进入轮廓拼接、区域构造，这块会成为最敏感模块之一。

### 6.5 `Measure2d`

这个模块可以稍后加入，但它对算量方向非常关键。

建议未来包含：

- 长度
- 周长
- 面积
- 质心

这会是从“几何内核”走向“算量内核”的第一步。

## 7. 结果结构体设计

建议库里建立统一规则：

- 简单纯数值运算可以返回标量
- 复杂判断和复杂运算返回结果结构体

典型对象包括：

- `ProjectionResult`
- `SegmentIntersectionResult`
- 后续的 `ContainmentResult`
- 后续的 `PolygonValidationResult`

结果结构体的好处是：

- 便于扩展字段
- 不破坏调用方
- 能保留过程语义

这点对工程软件非常重要，因为工程软件后面经常要追踪“为什么算成这样”。

## 8. 容差策略

这是整个设计里最需要尽早定下来的部分。

建议区分三层：

### 8.1 默认容差

库提供默认容差，例如：

- `kDefaultEpsilon`

用于大多数通用判断。

### 8.2 调用级可传入容差

关键判断类 API 应允许调用方传入 `eps`。

因为未来不同业务场景的几何尺度可能不同：

- 毫米级构件
- 米级平面图
- 导入图纸后的归一化坐标

容差需求未必一样。

### 8.3 业务层不直接散落容差

未来上层算量模块如果有业务容差，也应收敛到统一入口，不要在业务代码里到处写 magic number。

## 9. 面向算量软件的演进路线

你做这个库的最终目的，是给自己的算量软件打基础。所以设计上最好提前考虑“几何库如何一步步长成算量基础设施”。

建议分五个阶段。

### 阶段一：基础几何内核

内容：

- 点、向量、线段、包围盒
- 容差
- 距离、投影、相交

目标：

- 形成可信的 2D 核心算法底座

### 阶段二：轮廓和区域基础

内容：

- 折线
- 多边形
- 面积、周长、方向
- 点在多边形内判断

目标：

- 让图形可以表达“区域”

当前执行落点：

- 阶段二的路径层由 `polyline-design.md` 收敛
- 阶段二的区域层由 `polygon-design.md` 收敛
- 阶段二实现必须复用 `segment-design.md` 与 `box-design.md` 里已经定好的边段和包围盒规则

### 阶段三：图形清洗与拓扑预处理

内容：

- 去重
- 合并短边
- 闭合修复
- 自交检测
- 共线段整理

目标：

- 让脏数据进入可计算状态

### 阶段四：算量需要的几何组合能力

内容：

- 轮廓裁剪
- 区域布尔预备能力
- 偏移
- 构件截面计算辅助

目标：

- 能支撑实际工程量规则

### 阶段五：走向 3D 或参数化构件

内容：

- `Point3<T>`
- `Vector3<T>`
- 平面、空间线
- 3D 投影与距离

目标：

- 为更复杂构件和空间算量预留能力

## 10. 测试设计原则

虽然你现在说先不写代码，但设计阶段就应该把测试思想定下来，否则后面很容易写成“能跑但不可信”的库。

每个算法模块测试都建议覆盖四类情况：

- 正常情况
- 边界情况
- 退化情况
- 容差敏感情况

例如线段求交，至少要覆盖：

- 普通交叉
- 端点接触
- 平行不交
- 共线重叠
- 共线但分离
- 极短线段
- 近似平行

以后你会发现，真正难的不是“写出算法”，而是“证明这个算法在工程数据里稳定”。

如果只从设计角度看，现阶段最应该先拍板的是下面几件事。

是否采用：

- `Point2<T>` / `Vector2<T>` / `Segment2<T>` / `Box2<T>`

以及是否保留：

- `Point2d` / `Point2i`
- `Vector2d` / `Vector2i`

我建议保留这套方案。

### 11.2 容差接口方案

是否统一采用：

- 默认容差
- 关键 API 可传容差

我建议一定要统一，不然后面会反复返工。

### 11.3 第一阶段模块边界

我建议第一阶段只做：

- `types`
- `Predicate2d`
- `Distance2d`
- `Project2d`
- `Intersect2d`

先不碰：

- `Polyline`
- `Polygon`
- 布尔运算
- 3D

当前文档闭环：

- `types` 层中点和向量的规则已由 `point-vector-design.md` 固定
- `types` 层中线段的规则已由 `segment-design.md` 固定
- `types` 层中包围盒的规则已由 `box-design.md` 固定
- `types` 层中连续线的规则已由 `polyline-design.md` 固定
- `types` 层中多边形的规则已由 `polygon-design.md` 固定

### 11.4 结果结构体规则

建议从第一天开始就统一，否则后面接口风格会很乱。

## 12. 建议的第一版里程碑

建议你把“第一版几何算法库”定义成下面这个范围：

1. 完成基础类型设计
2. 完成容差体系设计
3. 完成几何谓词设计
4. 完成距离、投影、相交接口设计
5. 为每个模块定义测试清单

做到这一步，你就已经不是在“写几个几何函数”，而是在搭一个真正可演进的工程几何内核。

## 13. 结论

这个库最重要的不是一开始支持多少对象，而是先建立一套稳定规则：

- 类型怎么定义
- 算法怎么分层
- 容差怎么统一
- 结果怎么返回
- 测试怎么覆盖

如果这几件事先定好，后面你再扩展到多边形、图形清洗、区域计算、算量规则，整个工程会顺很多。

如果这几件事现在不定，后面代码一多就会很难收拾。

## 14. 向量方法设计规则

这一节用于明确 `Vec2Storage<T>`、`Point2<T>`、`Vector2<T>` 在方法层面的边界，尤其是常用向量方法该放在哪里。

### 14.1 `Vec2Storage<T>` 只做共享存储

`Vec2Storage<T>` 的职责限定为：

- 保存 `x`、`y`
- 提供最基础的构造和数据访问能力

不应在这一层放入：

- 容差判断
- 点和向量混合语义运算
- 投影、距离、相交等几何算法

原因是 `Vec2Storage<T>` 是底层共享实现，它应该保持中性，不能携带过多几何语义。

### 14.2 `Point2<T>` 保持轻量

`Point2<T>` 代表位置，不是普通二维数值容器。

建议支持的行为：

- 默认构造
- `(x, y)` 构造
- 与 `Vector2<T>` 的平移关系运算
- 严格相等比较

不建议支持的行为：

- `Point + Point`
- `Point * scalar`
- `Point / scalar`
- `Length()`
- `Normalized()`

### 14.3 `Vector2<T>` 可以承载纯向量能力

`Vector2<T>` 代表方向和位移，因此可以承载一部分纯向量方法。

建议支持：

- 默认构造
- `(x, y)` 构造
- 向量加减
- 标量乘除
- 一元负号
- `LengthSquared()`
- `Length()`

这些方法都属于向量本身的自然属性，放在 `Vector2<T>` 中是合理的。

### 14.4 `IsZero()` 必须走容差规则

`IsZero()` 虽然非常常用，但它不是普通成员属性，而是“带规则的几何判断”。

因此不建议把它直接做成 `Vector2<T>` 成员函数，原因如下：

- 浮点类型和整数类型的零判断规则不同
- 容差应由库统一控制，而不是藏在对象内部
- 后续还会有 `IsEqual`、`IsParallel`、`IsPointOnSegment` 等一系列判断，风格应保持统一

建议规则：

- 标量零判断放在谓词模块
- 向量零判断也放在谓词模块
- 调用时显式传入或使用默认容差

推荐风格：

- `IsZero(value, eps)`
- `IsZero(vector, eps)`

### 14.5 `Dot()` 和 `Cross()` 作为自由函数更稳妥

`Dot()` 和 `Cross()` 也是高频接口，但它们描述的是两个向量之间的关系，而不是单个对象自身的属性。

因此更推荐放在算法层或谓词层，作为自由函数存在。

推荐原因：

- 风格与 `Distance(a, b)`、`Intersect(a, b)` 更一致
- 不会让对象类型变成“万能工具类”
- 便于后续统一组织 2D 与 3D 几何算法接口

推荐风格：

- `Dot(lhs, rhs)`
- `Cross(lhs, rhs)`

### 14.6 `Normalized()` 不应静默处理零向量

归一化是常用操作，但它必然牵涉一个核心问题：零向量如何处理。

对工程几何库来说，不建议把零向量归一化后静默返回零向量，因为这会掩盖退化输入，后续在投影、偏移、方向判断中容易传播错误。

不建议的策略：

- 零向量直接返回零向量

更稳妥的策略是把“能否归一化”显式表达出来。

### 14.7 零向量判定规则

建议区分整数类型和浮点类型。

对于浮点向量：

- 零向量应按容差判断
- 本质上判断的是“是否接近零向量”

对于整数向量：

- 可按严格零判断

在几何算法主路径中，建议主要围绕 `Vector2d` 建立容差判断体系。

### 14.8 归一化接口建议

建议第一版优先提供：

- `TryNormalize(vector, normalized, eps)`

语义：

- 若向量接近零向量，则返回失败
- 若归一化成功，则输出单位向量

这比直接提供成员版 `Normalized()` 更安全。

第一版建议：

- 提供 `LengthSquared()`
- 提供 `Length()`
- 提供 `IsZero(vector, eps)`
- 提供 `TryNormalize(vector, normalized, eps)`
- 暂不强制提供成员版 `Normalized()`

### 14.9 当前建议的最终边界

当前阶段建议采用以下边界：

- `Vec2Storage<T>`：只放共享坐标存储
- `Point2<T>`：只放点语义相关的轻量能力
- `Vector2<T>`：放纯向量基础方法，如 `Length()`、`LengthSquared()`
- 谓词或算法层：放 `IsZero()`、`Dot()`、`Cross()`、`TryNormalize()`

这样做的核心目标不是追求“最少打字”，而是保证几何语义稳定、容差规则统一、后续接口可扩展。

## 15. 运算符设计规则

这一节用于明确 `Point2<T>` 和 `Vector2<T>` 应支持哪些运算符，哪些运算符应明确禁止。

设计原则只有一条：

- 运算符必须反映真实几何语义

不能因为 C++ 支持重载，就把所有看起来方便的运算都开放出来。

### 15.1 `Point2<T>` 的运算符边界

`Point2<T>` 表示位置，因此它不是普通二维数值向量。

建议支持的运算：

- `Point + Vector -> Point`
- `Point - Vector -> Point`
- `Point - Point -> Vector`
- `==`
- `!=`

这些运算都符合几何直觉：

- 一个点沿向量平移后，仍然是点
- 两个点相减，得到位移向量

建议明确不支持的运算：

- `Point + Point`
- `Point * scalar`
- `Point / scalar`
- 一元负号

原因：

- `Point + Point` 通常没有明确几何意义
- 点不是长度量，不适合直接缩放
- 一元负号会把点错误地当作原点对称的普通数值对象

### 15.2 `Vector2<T>` 的运算符边界

`Vector2<T>` 表示方向和位移，可以承载常见线性运算。

建议支持的运算：

- `Vector + Vector -> Vector`
- `Vector - Vector -> Vector`
- `Vector * scalar -> Vector`
- `Vector / scalar -> Vector`
- `scalar * Vector -> Vector`
- 一元负号
- `==`
- `!=`

这些运算都属于标准向量行为。

### 15.3 暂不建议直接支持的运算

以下接口建议暂时不要做成运算符或成员方法：

- `Vector + Point`
- `Dot` 成员函数
- `Cross` 成员函数
- 带容差的 `operator==`

原因如下。

对于 `Vector + Point`：

- 它与 `Point + Vector` 在语义上等价
- 只保留 `Point + Vector` 即可，接口更克制

对于 `Dot` 和 `Cross` 成员函数：

- 它们描述的是两个对象之间的关系
- 更适合统一作为自由函数放在算法层或谓词层

对于带容差的 `operator==`：

- 它会隐藏比较规则
- 阅读代码时很难判断“这是严格相等还是近似相等”

### 15.4 严格比较与容差比较分离

建议统一规则如下：

- `operator==` / `operator!=` 表示严格分量比较
- 容差比较使用单独接口

例如未来可采用：

- `IsEqual(lhs, rhs, eps)`

这样做的好处是：

- 代码语义清楚
- 调用方明确知道自己在做近似比较
- 容差策略不会被偷偷嵌入运算符

### 15.5 当前建议的最终规则

当前阶段建议采用如下约束：

`Point2<T>` 允许：

- `Point + Vector -> Point`
- `Point - Vector -> Point`
- `Point - Point -> Vector`
- `==`
- `!=`

`Point2<T>` 不允许：

- `Point + Point`
- `Point * scalar`
- `Point / scalar`
- 一元负号

`Vector2<T>` 允许：

- `Vector + Vector -> Vector`
- `Vector - Vector -> Vector`
- `Vector * scalar -> Vector`
- `Vector / scalar -> Vector`
- `scalar * Vector -> Vector`
- 一元负号
- `==`
- `!=`

容差判断统一放外部谓词或算法模块，不通过运算符隐式表达。
