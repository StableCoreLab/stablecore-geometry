# Polyline2 开发设计文档

## 1. 文档目的

本文档用于定义几何库中二维连续线 `Polyline2<T>` 的开发方案。后续开发人员必须按本文档实现，不再按临时理解扩展。

本文档覆盖以下内容：

- `Polyline2<T>` 的类型定位
- 连续性与闭合性规则
- 与 `Segment2<T>` 的关系
- 公共接口
- 参数化与累计长度规则
- 有效性规则
- 第一版实现范围

本文档是 `library-design.md`、`segment-design.md`、`box-design.md` 在连续线路径层的实现依据。

## 1.1 文档关系

本文档与其它设计文档的关系如下：

- `library-design.md` 负责定义折线和区域属于第二阶段能力，而连续线是进入多边形之前必须先稳定的一层路径抽象
- `segment-design.md` 负责定义 `Segment2<T>` 的边段语义、方向语义、长度语义、包围盒语义和参数化规则，而 `Polyline2<T>` 建立在有序边段序列之上
- `box-design.md` 负责定义 `Box2<T>` 的包围盒语义，而 `Polyline2<T>::GetBoundingBox()` 必须严格遵守这套语义

执行要求：

- 连续线文档不得另起一套边段方向或参数规则
- 连续线的整体长度与取点语义必须与单段语义兼容
- 若未来修改边段或包围盒规则，必须同步检查连续线文档

## 2. 设计定位

`Polyline2<T>` 表示由有序边段序列组成的二维连续路径对象。

这里的“连续线”不是单纯的“边段数组”，而是满足连接约束的路径。

它的目标是服务以下场景：

- 构件轮廓路径表达
- 开放路径表达
- 闭合轮廓的前置表达
- 路径长度统计
- 沿路径取点
- 后续多边形、轮廓拼接、图形清洗的基础数据结构

因此第一版必须同时保证：

- 对边段序列有明确顺序语义
- 对连续性有明确约束
- 对闭合性有明确表达
- 对整体参数化有统一规则

## 3. 总体设计结论

第一版统一按以下命名实现：

```cpp
template <typename T>
class Polyline2;

using Polyline2d = Polyline2<double>;
using Polyline2i = Polyline2<int>;
```

执行结论：

- `Polyline2<T>` 是 `types` 层路径对象
- 它不是多边形
- 它可以表示开放路径，也可以表示闭合路径
- 是否闭合是对象语义的一部分，不能靠调用方临时猜测

## 4. 为什么要先设计连续线，而不是直接设计多边形

多边形不是“很多边段的容器”，而是“满足更强约束的闭合连续线”。

如果跳过连续线直接设计多边形，后续会立刻混在一起的内容包括：

- 边段连接规则
- 闭合规则
- 顶点去重
- 自交规则
- 外环与洞
- 面积和方向

这些问题复杂度明显高于连续线本身。

因此更稳妥的工程顺序是：

1. 先定义边段
2. 再定义连续线
3. 最后在连续线基础上定义多边形

## 5. 类型语义与职责边界

### 5.1 `Polyline2<T>` 的核心语义

`Polyline2<T>` 表示按顺序连接起来的一组 `Segment2<T>`。

这意味着：

- 边段顺序有意义
- 路径方向有意义
- 头尾位置有意义
- 相同几何形状但反向遍历的路径，不是同一有向路径对象

### 5.2 `Polyline2<T>` 负责什么

`Polyline2<T>` 负责：

- 保存有序边段序列
- 对外提供路径顶点访问语义
- 表达开放/闭合路径状态
- 校验边段之间的连续关系
- 提供整体长度
- 提供整体包围盒
- 提供沿路径的整体参数取点

### 5.3 `Polyline2<T>` 不负责什么

`Polyline2<T>` 不负责：

- 多边形面积语义
- 外环/洞语义
- 布尔运算
- 图形清洗
- 自动拼接
- 自交修复
- 业务构件含义

## 6. 路径组成规则

### 6.1 基础组成单元

`Polyline2<T>` 的组成单元统一为：

- `Segment2<T>`

因此第一版路径允许混合以下边段：

- `LineSegment2<T>`
- `ArcSegment2<T>`

### 6.2 段序列顺序不可丢失

路径中的边段必须保留插入顺序。

不允许把路径视为无序边集合。

### 6.3 至少一个边段

第一版允许构造空路径对象，但：

- 空路径必须判为无效路径

原因：

- 空对象可以作为工程流程中的初始状态
- 但它不能伪装成有效连续线

## 7. 连续性规则

### 7.1 什么叫连续

对路径中任意相邻边段 `segments[i]` 与 `segments[i + 1]`，必须满足：

- 前一段终点与后一段起点相同，或在统一容差规则下近似相同

### 7.2 连续性是有效性的组成部分

只要出现某一对相邻边段不连续：

- 整条 `Polyline2<T>` 就必须判为无效

### 7.3 连续性按几何点判断，不按对象身份判断

对 `Polyline2<T>` 来说，相邻边段是否连续，判断依据是：

- 前一段的 `GetEndPoint()`
- 后一段的 `GetStartPoint()`

这两个点在几何上相同，或在统一容差下近似相同。

执行要求：

- 不要求两段共享同一个 `Point2<T>` 对象实例
- 不要求路径内部维护“共享点对象”来表达连续性
- 当前版本连续性只基于几何位置语义，不基于对象身份语义

### 7.4 不允许自动修正连接关系

第一版不允许在类型内部偷偷做以下操作：

- 自动反转某一段方向
- 自动移动端点
- 自动插入连接段
- 自动删除断裂段

执行结论：

- 路径对象负责暴露连续性问题
- 路径对象不负责偷偷修正连续性问题

## 8. 闭合性规则

### 8.1 闭合是对象语义

`Polyline2<T>` 必须显式表达路径是否闭合。

建议定义：

```cpp
enum class PolylineClosure
{
    Open,
    Closed
};
```

### 8.2 为什么闭合必须显式表达

因为：

- 有些路径首尾恰好重合，但语义上仍是开放路径
- 有些路径要求作为轮廓参与后续面积和区域构造
- 是否闭合会直接影响路径有效性检查和后续多边形构造

因此不能只靠“首尾点是否重合”去猜语义。

### 8.3 开放路径规则

若路径声明为 `Open`：

- 只要求相邻段连续
- 不要求最后一段终点与第一段起点相接

### 8.4 闭合路径规则

若路径声明为 `Closed`：

- 必须满足相邻段连续
- 还必须满足最后一段终点与第一段起点相同，或在统一容差下近似相同

执行要求：

- 闭合语义只由首尾边段连接关系表达
- 当前版本不额外存储一个“重复起点”作为独立顶点
- 闭合路径的几何闭合由边段关系保证，而不是靠额外顶点补丁保证

### 8.5 闭合路径不是多边形

即使路径是闭合的，也不代表它自动成为多边形。

因为它还未解决：

- 自交
- 面积方向
- 外环/洞
- 区域内部定义

## 9. 核心数据建议

第一版 `Polyline2<T>` 必须至少保存：

- 有序边段序列
- 闭合状态

建议形式：

```cpp
template <typename T>
class Polyline2
{
    std::vector<std::shared_ptr<Segment2<T>>> segments_;
    PolylineClosure closure_{PolylineClosure::Open};
};
```

这里的存储形式只是方向示意，不强制要求一定用 `shared_ptr`。

文档真正约束的是：

- 必须能承载异构边段
- 必须保留顺序
- 必须保留闭合状态

当前版本执行结论：

- `Polyline2<T>` 必须采用拥有式异构边段容器
- 调用方把边段交给路径对象后，路径对象必须拥有这些边段的生命周期
- 当前版本不采用“只保存外部引用”的非拥有式路径表示

这样做的原因是：

- 路径对象本身是稳定几何对象，不应依赖外部临时边段存活
- 后续轮廓、区域和算量流程都需要稳定持有路径

## 10. 公共接口

`Polyline2<T>` 第一版必须至少提供以下接口。

### 10.1 段访问接口

- `GetSegmentCount()`
- `GetSegment(index)`

### 10.2 路径状态接口

- `GetClosure()`
- `IsClosed()`

### 10.3 顶点接口

- `GetVertexCount()`
- `GetVertex(index)`

统一语义：

- 路径顶点表示路径连接点语义
- 开放路径的顶点数应为 `segmentCount + 1`
- 闭合路径的顶点访问不要求内部显式重复存储首顶点

执行要求：

- 顶点访问是路径语义接口，不代表内部必须以独立顶点对象为主存储
- 当前版本不引入独立 `Vertex2` 类型作为 `Polyline2<T>` 的核心数据

### 10.4 端点接口

- `GetStartPoint()`
- `GetEndPoint()`

统一语义：

- 开放路径的起点是第一段起点
- 开放路径的终点是最后一段终点
- 闭合路径的起点和终点在几何上应一致，但仍保留这两个接口

### 10.5 测量接口

- `Length()`

统一语义：

- 路径总长等于所有边段长度之和
- 路径总长必须与累计长度参数规则一致

### 10.6 包围盒接口

- `GetBoundingBox()`

统一语义：

- 返回所有边段包围盒合并后的最小轴对齐有效包围盒
- 返回语义必须遵守 `box-design.md`

### 10.7 参数取点接口

- `GetPointAt(parameter)`
- `GetPointAtLength(distanceFromStart, clampToPath = false)`
- `GetLengthAt(parameter)`
- `GetParameterAtLength(distanceFromStart, clampToPath = false)`

统一语义：

- `parameter ∈ [0, 1]`
- `GetPointAt(0)` 返回路径起点
- `GetPointAt(1)` 返回路径终点

转换接口语义：

- `GetLengthAt(parameter)` 把整体归一化参数转换为累计长度参数
- `GetParameterAtLength(distanceFromStart, clampToPath = false)` 把累计长度参数转换为整体归一化参数

### 10.8 有效性接口

- `IsValid()`

## 11. 整体参数化规则

### 11.1 统一参数

连续线整体继续沿用统一归一化参数：

- `t ∈ [0, 1]`

### 11.2 统一参数的几何含义

`Polyline2<T>` 的统一参数表示：

- 沿整条路径累计长度的归一化进度

也就是说：

- `t = 0.5` 表示走到整条路径总长度的一半位置

### 11.3 为什么整体参数必须按累计长度定义

如果按“段索引平均分配”定义，会立刻出现问题：

- 长短边段的参数密度失真
- 算量中的沿路径定位不可用
- 与 `Segment2<T>` 的长度参数体系不兼容

因此当前版本统一结论是：

- 路径整体参数按累计长度归一化

### 11.4 参数与长度的互转一致性

执行要求：

- 对有效路径，`GetLengthAt(0)` 必须返回 `0`
- 对有效路径，`GetLengthAt(1)` 必须返回 `Length()`
- 对有效路径，`GetParameterAtLength(0)` 必须返回 `0`
- 对有效路径，`GetParameterAtLength(Length())` 必须返回 `1`
- 对位于有效定义域内的参数，`GetPointAt(t)` 与 `GetPointAtLength(GetLengthAt(t))` 必须给出同一几何点

这条规则用于保证：

- 整体归一化参数
- 整体累计长度参数
- 单段参数体系

三者之间可追溯、可互转、可重建。

## 12. 累计长度参数规则

### 12.1 长度参数接口

连续线必须支持：

```cpp
GetPointAtLength(distanceFromStart, clampToPath = false)
```

### 12.2 统一语义

- `distanceFromStart = 0` 返回路径起点
- `distanceFromStart = Length()` 返回路径终点
- `distanceFromStart` 表示沿整条路径方向累计的带符号距离

### 12.3 越界规则

当 `clampToPath = true` 时：

- 返回点必须落在原路径上
- 小于零时返回路径起点
- 大于总长时返回路径终点

当 `clampToPath = false` 时：

- 允许外推
- 小于零时，从第一段的起点方向向前外推
- 大于总长时，从最后一段的终点方向向后外推

### 12.4 长度到参数的越界规则

当 `clampToPath = true` 时：

- `GetParameterAtLength(distanceFromStart, true)` 的返回值必须落在 `[0, 1]`
- 小于零时返回 `0`
- 大于总长时返回 `1`

当 `clampToPath = false` 时：

- `GetParameterAtLength(distanceFromStart, false)` 可以返回小于 `0` 或大于 `1` 的值
- 该值表示沿整条路径累计长度继续外推后的归一化位置
- 它必须与 `GetPointAtLength(distanceFromStart, false)` 保持可重建一致性

### 12.5 外推规则依赖边段语义

路径外推不能另起一套规则。

执行要求：

- 起点之前的外推必须调用第一段的外推语义
- 终点之后的外推必须调用最后一段的外推语义
- 不允许路径层自己发明和边段层冲突的外推逻辑

## 13. 有效性规则

`Polyline2<T>` 必须支持 `IsValid()`。

### 13.1 无效路径情况

以下情况必须判为无效：

- 没有任何边段
- 某一条边段自身无效
- 相邻边段不连续
- 声明为闭合路径但首尾不闭合

### 13.2 退化路径

第一版中，以下对象不视为有效路径：

- 总长度为零的路径

也就是说：

- 即使由若干退化边段组成，也不能伪装成有效连续线

### 13.3 闭合路径的端点语义

对有效闭合路径：

- `GetStartPoint()` 与 `GetEndPoint()` 必须返回几何上相同的点
- 但路径内部不要求额外存一个重复端点作为独立数据
- 闭合路径的“终点”只是为了保持与开放路径一致的接口形态

## 14. 包围盒规则

### 14.1 路径包围盒定义

路径包围盒必须由所有子边段的包围盒合并得到。

### 14.2 退化情况

若路径无效：

- `GetBoundingBox()` 的行为不应伪装成有效路径包围盒

当前文档只约束语义，不强制具体异常或断言策略。

## 15. 与多边形的关系

后续 `Polygon2<T>` 应建立在连续线之上，而不是绕过它直接建立在边段数组之上。

建议未来采用如下关系：

- `Polyline2<T>`：路径对象
- `Closed Polyline2<T>`：闭合路径对象
- `Polygon2<T>`：满足更强区域语义约束的闭合路径对象

执行结论：

- 多边形设计不得绕过连续线层

## 16. 第一版不做的内容

第一版 `Polyline2<T>` 不做以下内容：

- 自动拼接
- 顶点去重
- 闭合修复
- 自交修复
- 多边形面积语义
- 外环/洞
- 布尔运算
- 路径偏移

## 18. 开发验收标准

后续实现完成后，必须满足以下标准：

- `Polyline2<T>` 明确表示有序连续路径，而不是普通段数组
- 路径连续性规则清楚
- 开放/闭合语义清楚
- 路径整体参数化按累计长度归一化
- 路径长度参数与边段长度参数兼容
- 路径包围盒与 `Box2<T>` 语义一致
- 路径层不偷偷修复断裂、闭合或方向问题

## 19. 最终执行结论

后续 `Polyline2<T>` 开发必须按以下原则执行：

- `Polyline2<T>` 表示由有序 `Segment2<T>` 序列组成的连续路径
- 路径方向有意义，不能视为无序边集合
- 闭合性必须显式表达
- 连续性是有效性的组成部分
- 整体参数按累计长度归一化
- 整体长度参数与边段长度参数保持兼容
- 包围盒语义由 `box-design.md` 统一约束
- 多边形设计必须建立在连续线层之上

本文档即为 `Polyline2<T>` 的实现依据。
