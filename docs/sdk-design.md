# Geometry SDK 与二进制发布设计文档

## 1. 文档目的

本文档用于定义几何库未来作为 `DLL + import lib + headers` 发布时的边界规则。

本文档重点解决以下问题：

- 模板类型是否直接作为 DLL 稳定接口
- 哪些类型属于库内部泛型实现层
- 哪些类型属于对外 SDK 导出层
- 是否允许跨 DLL 边界暴露 STL 容器、虚类和模板类
- 如何在不推翻当前设计的前提下，为后续发布形态留出正确演进路径

本文档是 `library-design.md` 在“工程发布形态”和“ABI 边界”层的细化补充。

## 1.1 文档关系

本文档与其它设计文档的关系如下：

- `library-design.md` 负责定义整个几何库的分层、阶段目标和总边界
- `point-vector-design.md`、`segment-design.md`、`box-design.md`、`polyline-design.md`、`polygon-design.md` 负责定义几何语义本身
- 本文档负责定义这些几何语义未来如何安全地暴露为 SDK 和 DLL 接口

执行要求：

- 几何语义文档可以保留模板化设计
- 但任何面向外部程序的稳定二进制接口，都必须受本文档约束
- 若未来要发布正式 SDK，则必须先满足本文档中的导出边界要求，而不是直接把当前头文件内部实现整体暴露出去

## 2. 核心结论

当前库内部允许保留模板类型体系，但未来 DLL 对外接口不应直接以模板体系作为稳定 SDK 边界。

执行结论：

- 库内部可以继续存在 `Point2<T>`、`Segment2<T>`、`Box2<T>`、`Polyline2<T>`、`Polygon2<T>`
- 但 SDK 导出层第一版应统一收敛到 `double` 主路径
- 对外发布时，优先导出 `Point2d`、`Vector2d`、`LineSegment2d`、`ArcSegment2d`、`Box2d`、`Polyline2d`、`Polygon2d`
- 不把“任意模板实例都可直接跨 DLL 使用”作为设计目标

这条结论必须尽早固定，否则后续头文件组织、实现组织和 ABI 约束都会持续漂移。

## 3. 为什么模板类型不适合作为 DLL 稳定接口

模板类型本身并不是不能和 DLL 共存，但它不适合作为稳定对外二进制接口的主体。

原因如下：

- 模板通常要求调用方在编译期看到完整实现
- 这会迫使大量实现继续放在头文件里
- 一旦调用方自己实例化模板，就很难把实现真正收进 DLL
- 模板实例集合如果不受控，会导致导出边界膨胀
- 不同编译器、编译选项、STL 实现下，跨模块 ABI 更脆弱

因此如果目标明确是：

- 生成 DLL
- 给别人 `headers + lib + dll`
- 让别人以 SDK 方式接入

那就不能把“模板头文件自然外露”当成最终发布形态。

## 4. 当前推荐架构

未来发布形态统一分三层：

### 4.1 内部几何语义层

这一层允许保留当前模板设计：

- `Point2<T>`
- `Vector2<T>`
- `Segment2<T>`
- `LineSegment2<T>`
- `ArcSegment2<T>`
- `Box2<T>`
- `Polyline2<T>`
- `Polygon2<T>`

这一层的职责是：

- 验证几何抽象是否合理
- 支撑内部算法实现
- 保持类型系统表达力

这一层不是对外稳定 ABI 承诺层。

### 4.2 内部实现收敛层

这一层负责把模板实现收敛到受控实例。

第一版执行建议：

- 只把 `double` 作为正式算法主路径
- `int` 等其它模板实例只作为内部表达、测试或局部辅助
- 若某个类型确实需要编译进库体，则显式实例化 `double` 版本

这层的核心目标是：

- 防止模板实例无边界扩散
- 为 DLL 导出层准备单一精度主路径

### 4.3 SDK 导出层

这一层才是未来给外部程序的稳定接口层。

第一版执行结论：

- SDK 统一导出 `double` 专用几何类型
- SDK 不承诺任意模板参数版本
- SDK 头文件不要求调用方重新编译大量模板实现

建议导出类型：

- `Point2d`
- `Vector2d`
- `LineSegment2d`
- `ArcSegment2d`
- `Box2d`
- `Polyline2d`
- `Polygon2d`

## 5. 第一版 SDK 精度策略

第一版 SDK 统一采用 `double` 主路径。

执行结论：

- `double` 是对外正式发布精度
- `int` 不是 DLL 稳定接口承诺的一部分
- 若未来需要整数输入场景，应通过构造、转换或单独 API 支持，而不是直接把所有 `T` 暴露到 SDK 边界

这样做的原因是：

- 几何算法主路径本就更适合浮点计算
- ABI 更清晰
- 文档、测试和导出策略都更容易收敛
- 更符合未来算量产品主场景

## 6. 跨 DLL 边界的禁止项

为了保证 SDK 稳定性，跨 DLL 边界应尽量避免直接暴露以下内容：

- 任意模板类
- 未收敛实例的模板函数
- STL 容器作为长期稳定 ABI 承诺
- 依赖编译器布局细节的复杂模板嵌套类型
- 以内部虚继承层直接作为公开 ABI 主体

执行建议：

- 当前内部可以继续用 `std::vector`、`std::shared_ptr`、抽象基类和模板
- 但 SDK 层不要直接把这些内部组织方式当成对外稳定承诺

这意味着：

- 当前 `Polyline2<T>` 内部采用拥有式异构段容器是允许的
- 但未来 SDK 是否直接暴露该内部容器，不应现在就锁死

## 7. 对当前代码设计的影响

当前这套文档和代码设计不需要立刻推翻，但必须增加一个新的工程边界：

- “模板设计”不等于“未来 SDK 直接发布形态”

对现阶段的具体影响如下。

### 7.1 类型设计可以继续模板化

当前 `types` 层可以继续保持模板形式，用于把几何语义先设计和实现清楚。

### 7.2 算法实现应优先围绕 `double`

即使保留模板类型，算法主路径也应优先保证：

- `Point2d`
- `Segment2d`
- `Box2d`
- `Polyline2d`
- `Polygon2d`

这会直接降低未来 DLL 收敛成本。

### 7.3 文档不再把“所有模板实例都属于正式发布目标”作为默认前提

后续文档如果提到类型别名，应明确区分：

- 内部模板语义
- 对外 SDK 导出形态

## 8. 当前不建议立即做的事情

当前阶段不建议立即为了 DLL 发布而做以下大改：

- 立刻删除全部模板设计
- 立刻把所有类型改成纯 C API
- 立刻把所有内部容器和抽象层重写成 ABI 专用结构

原因是：

- 你现在仍处于几何语义和算法边界快速收敛阶段
- 过早为 ABI 做到底，会显著拖慢主线设计

当前更合理的做法是：

- 先把几何语义层做稳
- 同时从现在开始约束未来导出边界

## 9. 后续演进建议

未来从内部库走向正式 SDK，建议按以下顺序推进：

1. 保持内部模板层，但收敛算法主路径到 `double`
2. 为核心类型补一层受控导出接口
3. 明确哪些 API 属于 SDK，哪些仅属于内部实现
4. 在需要时再设计更稳定的 ABI 层，例如：
   - 非模板导出类
   - pImpl
   - handle API
   - C ABI 包装层

第一版不要求一步到位，但必须沿这个方向收敛，不能继续把模板头文件自然外露当成最终交付模式。

## 10. 当前执行结论

后续开发必须按以下原则执行：

- 内部可以保留模板化几何类型体系
- DLL 对外发布层不以模板体系作为稳定 ABI
- 第一版 SDK 导出统一收敛到 `double` 主路径
- 当前代码实现允许继续服务模板设计，但后续算法和接口优先围绕 `double` 收敛
- 未来若进入正式 SDK 阶段，必须单独设计导出层，而不是直接把当前内部头文件整体外发

本文档即为后续几何库发布形态与二进制接口设计依据。

## 11. 当前落地入口

当前仓库已经补齐一个最小的 SDK 入口层，建议外部调用方优先包含：

```cpp
#include "sdk/Geometry.h"
```

当前实际落地的第一版 SDK 面是：

- `geometry::sdk::GeoPoint2d`
- `geometry::sdk::GeoVector2d`
- `geometry::sdk::GeoBox2d`
- `geometry::sdk::ProjectionResult2d`
- `Distance`、`DistanceSquared`
- `ProjectPointToSegment`
- `Contains`、`Intersects` 的 box 查询入口

当前不把 `Polyline2d`、`Polygon2d` 作为 SDK 首发面的一部分，也不把内部多态段容器作为对外 ABI 承诺。
