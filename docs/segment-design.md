# 线段体系开发设计文档

## 1. 文档目的

本文档用于定义几何库中二维边段体系的开发方案。后续开发人员必须按本文档实现，不再按临时理解扩展。

本文档覆盖以下内容：

- 统一边段抽象
- 直线段与圆弧段的类型边界
- 数据不变量
- 公共接口
- 参数化规则
- 算法层依赖方式
- 第一版实现范围
- 面向算量场景的演进要求

本文档是 `library-design.md`、`point-vector-design.md` 与 `box-design.md` 在线段层的延续实现依据。

## 1.1 文档关系

本文档与其它设计文档的关系如下：

- `library-design.md` 负责定义 `Segment2<T>` 进入核心类型体系的全局边界，以及它与 `Box2<T>` 一起构成第一阶段核心类型
- `point-vector-design.md` 负责定义 `Point2<T>` 与 `Vector2<T>` 的语义边界，而线段端点、方向和长度相关表达建立在这套类型规则上
- `box-design.md` 负责定义 `Box2<T>` 的有效性、退化规则和闭区间包围盒语义，而 `Segment2<T>::GetBoundingBox()` 必须严格遵守这套语义

执行要求：

- 线段文档中的包围盒接口不得脱离 `box-design.md` 单独演化
- 若未来修改 `Box2<T>` 的边界规则，必须同步检查线段包围盒接口和圆弧包围盒规则
- 若未来修改边段参数化或端点语义，也必须同步检查包围盒相关算法说明是否需要联动调整

## 2. 设计定位

这里的“线段”不是狭义的数学直线段，而是二维轮廓中的“边段单元”。

对未来算量产品来说，边段体系最终需要支撑：

- 构件轮廓表达
- 开放路径与闭合轮廓表达
- 边界长度统计
- 投影、裁剪、求交、最近点
- 区域构造前的边界清洗与拼接

因此第一版必须同时考虑：

- 当前实现可控
- 后续扩展到轮廓与区域时接口不返工

## 3. 总体设计结论

二维边段体系采用“统一抽象 + 具体类型”的结构。

第一版统一按二维命名实现：

```cpp
enum class SegmentKind2
{
    Line,
    Arc
};

enum class ArcDirection
{
    Clockwise,
    CounterClockwise
};

template <typename T>
class Segment2;

template <typename T>
class LineSegment2;

template <typename T>
class ArcSegment2;
```

并提供别名：

```cpp
using Segment2d = Segment2<double>;
using Segment2i = Segment2<int>;

using LineSegment2d = LineSegment2<double>;
using LineSegment2i = LineSegment2<int>;

using ArcSegment2d = ArcSegment2<double>;
using ArcSegment2i = ArcSegment2<int>;
```

执行结论：

- 文档统一采用 `Segment2<T>` 命名
- 不再使用 `Segment<T>` 这种省略维度的信息命名
- 后续若进入三维，可自然扩展为 `Segment3<T>` 或其它更明确的类型体系

## 4. 为什么必须先统一边段抽象

统一边段抽象不是为了形式上的继承层次，而是为了保证后续工程结构稳定。

后续这些对象都会依赖“边段单元”抽象：

- 折线
- 轮廓
- 区域边界
- 构件轮廓边
- 算量边界线集合

如果不先统一抽象，后续会快速出现这些问题：

- 容器无法统一承载直线段和圆弧段
- 公共逻辑如端点、长度、包围盒被重复实现
- 算法接口被迫拆成多套
- 轮廓与区域层被具体类型细节污染

因此第一版必须先统一“二维边段”这一层语义。

## 5. 类型体系与职责边界

### 5.1 `Segment2<T>`

`Segment2<T>` 是统一边段抽象。

它的职责是：

- 对外表达“这是一个二维边段”
- 提供所有边段都必须具备的公共接口
- 作为算法层统一入口

它不负责：

- 存储具体直线段字段
- 存储具体圆弧段字段
- 承载求交、投影、裁剪等复杂算法实现
- 承载业务含义

实现约束：

- `Segment2<T>` 必须是抽象基类
- `Segment2<T>` 必须提供虚析构函数
- 公共接口应通过虚函数暴露，由派生类完成具体实现
- 基类不应存储任何派生类专属缓存

### 5.2 `LineSegment2<T>`

`LineSegment2<T>` 表示有限直线段。

它的职责是：

- 保存起点和终点
- 提供线性参数化
- 提供直线段的长度、方向、包围盒等基础行为

### 5.3 `ArcSegment2<T>`

`ArcSegment2<T>` 表示有限圆弧段。

它的职责是：

- 保存圆弧核心参数
- 提供弧参数化
- 提供弧长、起终点、包围盒等基础行为

第一版只支持圆弧，不支持：

- 椭圆弧
- 贝塞尔曲线
- 样条曲线
- 整圆边段
- 复合曲线段

## 6. 基类设计原则

`Segment2<T>` 必须是“薄抽象”，不允许发展成万能基类。

执行原则：

- 基类只定义公共语义接口
- 派生类负责保存各自最小必要数据
- 复杂算法统一留在 `algorithm` 层
- 算法层采用“统一抽象 + 显式分发”

不建议把投影、相交、偏移、裁剪全部做成虚函数塞进基类。原因是：

- 基类会快速膨胀
- 算法矩阵会变复杂
- 新类型接入成本会升高
- 测试边界变模糊

这对工程几何库尤其不利。

## 7. 统一公共接口

`Segment2<T>` 第一版必须至少提供以下公共接口。

### 7.1 类型接口

- `GetKind()`

返回：

- `SegmentKind2::Line`
- `SegmentKind2::Arc`

执行要求：

- 调用方不得依赖隐式假设猜测真实类型
- 需要分派时，显式读取 `GetKind()`
- `GetKind()` 的返回值必须与实际派生类型一一对应

### 7.2 端点接口

- `GetStartPoint()`
- `GetEndPoint()`

统一语义：

- 所有边段都必须有明确起点
- 所有边段都必须有明确终点
- 起点和终点有顺序语义，不是无序端点对

这对后续轮廓方向、偏移方向、闭合遍历都很关键。

执行要求：

- 起点和终点必须以几何定义为准，不允许做自动交换式规范化
- `LineSegment2<T>` 与 `ArcSegment2<T>` 都必须稳定返回自身定义方向上的起终点

### 7.3 长度接口

- `Length()`

第一版不强制统一 `LengthSquared()`。

原因：

- 直线段有自然意义
- 圆弧段并无同等直接语义收益
- 统一强推会造成无效接口一致性

执行要求：

- `Length()` 必须返回边段的几何长度
- `Length()` 的结果必须与 `GetPointAtLength(Length())` 返回终点语义一致
- 对无效边段，`Length()` 的行为由实现阶段统一约束，但不得伪装成有效长度

### 7.4 包围盒接口

- `GetBoundingBox()`

返回类型应统一为：

- `Box2<T>`

执行结论：

- `GetBoundingBox()` 当前版本统一返回 `Box2<T>`
- 若算法主路径需要浮点包围盒，应通过 `Segment2d` / `Box2d` 进入，而不是让接口本身返回类型摇摆

最终以 `Box2` 设计文档为准，但线段体系必须预留这一接口。

这条接口与 `box-design.md` 中 `Box2<T>` 的闭区间、退化盒有效、无效盒规则是同一套语义，不允许线段体系单独定义另一套包围盒解释。

执行要求：

- `GetBoundingBox()` 必须返回当前边段的最小轴对齐有效包围盒
- `LineSegment2<T>` 的包围盒由两个端点坐标直接确定
- `ArcSegment2<T>` 的包围盒必须覆盖整段圆弧，不允许只取端点外接矩形
- `GetBoundingBox()` 的返回值必须满足 `box-design.md` 中的有效盒规则

### 7.5 参数接口

- `GetPointAt(parameter)`
- `GetPointAtLength(distanceFromStart, clampToSegment = false)`

参数区间统一定义为：

- `parameter ∈ [0, 1]`

统一语义：

- `0` 对应起点
- `1` 对应终点

这条规则必须贯穿所有边段类型，不得因为具体类型不同而改变参数含义。

长度参数接口语义：

- `distanceFromStart = 0` 对应起点
- `distanceFromStart = Length()` 对应终点
- 默认 `clampToSegment = false`

该接口用于表达面向工程和算量的“距起点走了多远”。

### 7.6 有效性接口

- `IsValid()`

统一原则：

- 能构造出对象，不代表对象有效
- 无效对象必须能被显式识别
- 算法层可以在必要时拒绝无效对象

## 8. 统一禁止项

`Segment2<T>` 禁止承载以下内容：

- 直线段专属方向缓存
- 圆弧段专属圆心、半径、角度字段
- 求交算法
- 投影算法
- 最近点算法
- 偏移算法
- 与工程业务对象耦合的字段

禁止原因：

- 基类职责会失控
- 算法与数据边界会混乱
- 后续扩展其它曲线段会更难收敛

## 9. 线段几何语义规则

### 9.1 边段有方向

所有边段都必须视为有方向对象。

这意味着：

- `start -> end` 是有意义的
- 相同几何位置但反向的边段，不应视为同一有向对象
- 后续轮廓遍历、法向推导、偏移方向都会依赖这个定义

### 9.2 边段是有限对象

第一版只讨论有限边段。

不支持：

- 无限直线
- 射线
- 整圆
- 无限周期曲线

### 9.3 边段允许退化输入被构造，但不得被默认为有效

例如：

- 起终点重合的直线段
- 半径非法的圆弧段
- 无法形成有限弧长的圆弧段

都可以作为对象存在，但 `IsValid()` 必须返回失败。

这样做的原因是：

- 上层数据可能脏
- 先接收、再校验，比分散在构造阶段抛异常更利于工程流程控制

## 10. `LineSegment2<T>` 设计规则

### 10.1 核心数据

`LineSegment2<T>` 第一版必须至少保存：

- 起点 `Point2<T>`
- 终点 `Point2<T>`

第一版不要求缓存：

- 方向向量
- 长度
- 单位方向
- 包围盒

### 10.2 必须满足的规则

- 起点和终点必须可直接访问
- 起点和终点顺序不可丢失
- 零长度线段必须可识别为退化对象

### 10.3 可额外提供的专属接口

可提供但不进入统一基类的接口：

- `DirectionVector()`
- `MidPoint()`

是否进入第一版实现，可按代码阶段再定，但文档层允许存在。

### 10.4 参数化规则

对 `LineSegment2<T>`：

- `GetPointAt(t)` 采用线性插值

即：

- `t = 0` 返回起点
- `t = 1` 返回终点
- 中间值按端点线性插值

### 10.5 有效性规则

以下情况必须视为无效：

- 起点与终点严格重合，或在容差意义下接近重合

具体容差规则仍由统一谓词模块控制，不允许在类型内部散落 magic number。

## 11. `ArcSegment2<T>` 设计规则

### 11.1 核心数据

第一版 `ArcSegment2<T>` 采用以下主表达：

- 圆心 `Point2<T>`
- 半径
- 起始角
- 终止角
- 方向 `ArcDirection`

执行要求：

- 第一版只允许这一套主表达作为核心数据
- 不允许同时把“圆心半径角度”和“三点定弧”并行维护为双主表达

原因是双主表达很容易带来状态不一致。

### 11.2 圆弧方向规则

圆弧方向必须显式表达。

不允许通过：

- 起止角大小关系
- 参数正负
- 某个隐式约定

去推断方向。

执行要求：

- `ArcDirection` 必须是显式字段
- 任意算法都应直接读取方向，不做猜测

### 11.3 起终点规则

圆弧必须能稳定返回：

- 起点
- 终点

起点与终点由以下信息共同确定：

- 圆心
- 半径
- 起止角
- 方向

### 11.4 弧长规则

必须提供：

- `Length()`

弧长应由：

- 半径
- 有向扫角

共同决定。

执行要求：

- `ArcSegment2<T>` 的弧长必须是从起点到终点的有向路径长度
- 当 `clampToSegment = false` 时，`GetPointAtLength()` 的外推长度应沿相同扫角方向继续累计
- `Length()` 为零或接近零的圆弧必须判为无效

### 11.5 参数化规则

对 `ArcSegment2<T>`：

- `GetPointAt(0)` 返回起点
- `GetPointAt(1)` 返回终点
- 中间参数按有向扫角做映射

注意：

- 参数不是“角度值”
- 参数是统一边段参数
- 角度只是圆弧内部实现细节
- `GetPointAt(t)` 与 `GetPointAtLength(t * Length())` 在 `t` 位于 `[0, 1]` 时应给出同一几何点

### 11.6 第一版不支持整圆

即使起止角从数值上可以绕满一圈，第一版也不允许把整圆表达为单一 `ArcSegment2<T>`。

原因：

- 它没有唯一的起点终点语义
- 与统一边段接口冲突
- 会增加参数化和交点结果表达复杂度

## 12. 圆弧角度与归一化规则

为了避免后续实现分歧，第一版应统一以下规则。

### 12.1 角度单位

第一版统一使用弧度制。

不允许：

- 某些接口用角度制
- 某些接口用弧度制

### 12.2 角度归一化

文档建议内部统一支持角度归一化，但：

- 是否把角度强制规范到固定区间
- 规范到 `[0, 2π)` 还是 `(-π, π]`

不作为类型外部语义的一部分。

外部调用方不应依赖角度原始存储格式。

### 12.3 有向扫角

圆弧的真实几何跨度必须由：

- 起始角
- 终止角
- 方向

共同决定。

不能只看 `end - start` 的数值结果。

### 12.4 有效性要求

第一版应明确拒绝以下情况：

- 半径小于等于零
- 无法形成有限正弧长
- 起止角和方向组合无法确定唯一有限圆弧

## 13. 参数化统一规则

参数化是后续算法可扩展性的关键，必须提前统一。

### 13.1 统一参数区间

所有边段统一采用：

- `t ∈ [0, 1]`

### 13.2 统一端点语义

- `t = 0` 对应起点
- `t = 1` 对应终点

### 13.3 参数不等于长度比例

必须明确：

- 对直线段，参数与长度比例一致
- 对圆弧段，参数与弧长比例一致

第一版建议统一采用“归一化路径进度”语义。

也就是说：

- `t = 0.5` 应表示沿该边段走了一半的路径长度

对直线段这与线性插值一致。

对圆弧段这意味着：

- `t` 应映射到扫角的一半
- 因为圆弧曲率恒定，所以它也等价于弧长进度的一半

### 13.4 参数超界规则

第一版应在文档中明确：

- `GetPointAt(t)` 的正式定义域是 `[0, 1]`

对于越界行为，建议采用以下策略之一，但必须全库统一：

- 调试期断言，发布期由调用方保证
- 进行显式裁剪
- 定义为未指定行为

当前建议：

- 将 `[0, 1]` 视为调用前置条件
- 类型层不偷偷裁剪参数

这样更利于暴露错误调用。

### 13.5 长度参数规则

除统一归一化参数外，边段体系应同时支持长度参数。

建议统一接口：

```cpp
GetPointAtLength(distanceFromStart, clampToSegment = false)
```

统一语义：

- `distanceFromStart` 表示从起点沿边段方向累计的带符号距离
- `distanceFromStart = 0` 返回起点
- `distanceFromStart = Length()` 返回终点
- 默认 `clampToSegment = false`

执行要求：

- `distanceFromStart` 的正方向必须与边段定义方向一致
- 该参数不是“绝对值距离”，允许负值表达向起点前方外推
- 对 `LineSegment2<T>` 与 `ArcSegment2<T>`，`GetPointAtLength()` 必须保持一致的方向定义

执行要求：

- `distanceFromStart` 的正方向必须与边段定义方向一致
- 该参数不是绝对值距离，允许负值表达向起点前方外推
- 对 `LineSegment2<T>` 与 `ArcSegment2<T>`，`GetPointAtLength()` 必须保持一致的方向定义

### 13.6 长度参数越界语义

当 `clampToSegment = true` 时：

- 返回点必须落在该边段上
- `distanceFromStart < 0` 时返回起点
- `distanceFromStart > Length()` 时返回终点

当 `clampToSegment = false` 时：

- 允许沿边段参数方向继续外推
- `distanceFromStart < 0` 时，沿起点之前的方向延伸
- `distanceFromStart > Length()` 时，沿终点之后的方向延伸

这条规则用于服务算量中常见的按距离找点、偏移定位、构件分段定位等场景。

### 13.7 不同边段类型的外推规则

当 `clampToSegment = false` 时，不同边段类型的行为必须显式约束。

对 `LineSegment2<T>`：

- 长度参数越界后，按所在直线方向延伸
- 负距离表示从起点沿反方向延伸
- 超过总长度表示从终点沿正方向延伸

对 `ArcSegment2<T>`：

- 长度参数越界后，按同一圆、同一半径、同一旋转方向继续延伸
- 负距离表示在起点之前沿圆弧反向回退
- 超过总长度表示在终点之后沿圆弧方向继续前进

执行要求：

- 不允许对直线段和圆弧段采用不同风格的长度参数解释
- 外推后的点可以不属于原有限边段，但必须属于该边段对应的基础几何支撑对象

### 13.8 为什么长度参数不替代统一参数

长度参数虽然更贴近算量，但不能替代统一参数 `t ∈ [0, 1]`。

原因：

- 统一参数更适合结果结构体表达
- 统一参数更适合跨类型算法共享接口
- 长度参数会随边段总长变化，不适合作为统一归一化描述

因此执行结论是：

- `GetPointAt(t)` 保留为统一参数接口
- `GetPointAtLength(distanceFromStart, clampToSegment = false)` 作为工程长度参数接口并存

## 14. 有效性规则

所有边段都必须支持 `IsValid()`。

统一原则：

- 对象存在不代表有效
- 算法可以显式拒绝无效对象
- 退化对象不得伪装成普通有效边段
- 有效性判定必须与 `point-vector-design.md` 和 `box-design.md` 的数值规则保持一致，不允许各自定义一套阈值

### 14.1 `LineSegment2<T>` 无效条件

- 起点与终点重合或近似重合

### 14.2 `ArcSegment2<T>` 无效条件

- 半径非法
- 起止角非法
- 方向缺失或不合法
- 无法构成有限正弧长
- 起点与终点语义不明确

### 14.3 容差边界

具体“近似重合”“非法半径”“零弧长”等判断规则，统一交给谓词或数值模块。

执行要求：

- 类型层不散落 `1e-9`
- 不能在某个类型内部私自定义一套弧段容差

## 15. 包围盒设计约束

边段体系必须预留统一包围盒接口，因为后续以下功能都会依赖它：

- 粗判求交
- 空间索引
- 候选集过滤
- 轮廓拼接预筛

### 15.1 直线段包围盒

应由起终点坐标直接确定。

### 15.2 圆弧段包围盒

不能只由起点和终点决定。

必须考虑：

- 圆弧是否跨越极值角

这条规则非常关键，否则很多弧段包围盒会偏小，进而导致粗判漏检。

### 15.3 第一版执行要求

即使 `Box2<T>` 还未正式写出，线段文档也必须先确定：

- 边段包围盒是强制能力
- 圆弧包围盒必须是几何正确包围盒，而不是端点包围盒

当前文档闭环：

- `GetBoundingBox()` 的返回语义以 `box-design.md` 为准
- `Box2<T>` 对退化盒和无效盒的解释会直接影响边段粗判与过滤行为
- `segment-design.md` 与 `box-design.md` 对圆弧包围盒的描述必须始终保持一致

## 16. 算法层如何依赖边段体系

算法层依赖边段体系时，必须分为两层。

### 16.1 公共层算法

只依赖 `Segment2<T>` 公共接口，例如：

- 读取端点
- 读取长度
- 读取包围盒
- 读取参数点
- 校验有效性

### 16.2 具体类型算法

需要几何细节时，先分派类型，再进入具体实现，例如：

- 直线段与直线段求交
- 直线段与圆弧段求交
- 圆弧段与圆弧段求交
- 点到边段投影
- 点到边段最近点

执行原则：

- 不要求所有算法都依赖虚函数多态完成
- 明确允许“统一抽象 + 显式类型分发”

这更符合工程几何库的复杂算法组织方式。

## 17. 第一版应同步定义的结果结构体

为了服务后续求交、投影、算量规则，线段体系设计阶段应同步约束结果表达。

虽然本文件不直接定义算法，但应先确定结果风格。

### 17.1 投影结果

建议后续定义：

```cpp
template <typename T>
struct SegmentProjectionResult2;
```

第一版必须至少包含：

- 是否成功
- 最近点
- 参数值
- 距离平方
- 是否落在边段内部
- 投影所落边段类型标识
- 是否发生外推

执行要求：

- 参数值必须同时支持归一化参数和长度参数的可追溯转换，至少保留其中一种作为主参数
- 最近点必须满足 `GetPointAt(parameter)` 或 `GetPointAtLength(distanceFromStart)` 的可重建性

### 17.2 求交结果

建议后续定义：

```cpp
enum class SegmentIntersectionKind2
{
    None,
    Point,
    Overlap,
    Tangent,
    Parallel,
    CollinearSeparate
};

template <typename T>
struct SegmentIntersectionResult2;
```

最终枚举值可在求交文档中再收敛，但本阶段必须先明确：

- 复杂算法不能只返回 `bool`
- 必须保留过程语义

第一版求交结果至少应包含：

- 结果种类
- 交点集合或唯一交点
- 双方参数值
- 是否重叠
- 是否共线
- 是否平行
- 是否存在端点接触

执行要求：

- 参数值必须能追溯到两条边段各自的参数体系
- 若结果为重叠或共线，必须显式表达，而不是塞进一个普通交点结构里

### 17.3 结果结构设计原则

所有复杂结果结构体应遵循：

- 显式表达状态
- 显式表达参数位置
- 显式表达几何输出
- 允许未来扩字段而不破坏调用方

这对算量产品尤其重要，因为上层经常要追溯“为什么判断成这样”。

## 18. 与轮廓层的接口约束

线段体系不是孤立设计，必须提前考虑它如何进入 `Polyline2` 或轮廓边界层。

### 18.1 轮廓层最关心的边段能力

- 起点
- 终点
- 长度
- 包围盒
- 有向性
- 参数化

### 18.2 轮廓拼接要求

未来若做轮廓拼接，边段必须支持：

- 明确首尾连接关系
- 区分正向与反向
- 区分几何重合与方向一致性

因此第一版线段设计中：

- “有方向”
- “起终点有顺序”
- “统一参数 `0 -> 1`”

这些都不能模糊。

### 18.3 面向算量的意义

算量产品里很多边界并不是纯展示图元，而是：

- 轮廓闭合的边
- 构件截面边
- 需要统计长度和拼接关系的边

所以线段类型必须优先服务“稳定可算”，而不是优先服务“画图方便”。

## 19. 第一版不做的内容

第一版边段体系不做以下内容：

- 样条曲线段
- 椭圆弧段
- 贝塞尔曲线段
- 整圆边段
- 复合曲线段
- 偏移边段
- 业务层构件边
- 自动拼接逻辑
- 拓扑清洗逻辑

第一版范围严格限制为：

- 有限直线段
- 有限圆弧段
- 统一二维边段抽象

## 21. 开发验收标准

后续实现完成后，必须满足以下标准：

- 直线段与圆弧段可通过统一抽象使用
- 调用方可以可靠区分边段类型
- 所有边段都具备统一起点、终点、长度、包围盒、参数点、有效性接口
- 参数区间统一为 `[0, 1]`
- 圆弧方向是显式字段，不靠隐式猜测
- 圆弧包围盒是几何正确包围盒
- 算法层可先基于统一接口工作，再按类型细分
- 第一版只支持直线段和有限圆弧段

## 22. 最终执行结论

后续线段体系开发必须按以下原则执行：

- 统一采用 `Segment2<T>`、`LineSegment2<T>`、`ArcSegment2<T>` 命名
- 建立统一二维边段抽象
- 统一支持有限直线段与有限圆弧段
- 统一参数区间为 `[0, 1]`
- 统一提供起点、终点、长度、包围盒、参数点、有效性接口
- 明确边段有方向
- 复杂算法留在 `algorithm` 层
- 算法层采用“统一抽象 + 显式分发”模式
- 结果结构体优先保留过程语义
- 设计优先服务后续轮廓与算量场景，而不是只服务当前演示实现

文档闭环要求：

- 全局模块边界以 `library-design.md` 为总入口
- 点和向量语义边界以 `point-vector-design.md` 为准
- 包围盒返回语义与退化规则以 `box-design.md` 为准
- 任何涉及 `Segment2<T>` 包围盒语义的实现或文档修改，都必须同步检查上述文档是否需要联动调整

本文档即为线段体系的实现依据。
